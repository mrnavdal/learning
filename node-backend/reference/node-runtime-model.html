Node runtime model

Zhuštěná esence. K rychlému opakování před pohovorem.

Model v jedné větě

Jedno vlákno na tvůj JavaScript + event loop, který žongluje s čekáním na I/O, + malý libuv thread pool (výchozí 4) na pár těžkých operací na pozadí.

Proč to škáluje

  • Server tráví většinu času čekáním (DB, disk, síť), ne počítáním.
  • Neblokující operace = vlákno se během čekání uvolní a obsluhuje další požadavky.
  • Callback/promise se vyřídí, až data dorazí.

Blokující vs. neblokující

Blokující ❌Neblokující ✅
fs.readFileSync()await fs.promises.readFile()
crypto.pbkdf2Sync()await db.query(), await fetch()
Cokoliv s …SyncCokoliv s callbackem / promise

Pravidlo: na serveru se …Sync a těžkému CPU v hlavním vlákně vyhýbáš — zablokuješ event loop a čekají všichni.

libuv thread pool

  • Výchozí 4 vlákna (UV_THREADPOOL_SIZE), sdílená — ne vlákno na požadavek.
  • Používá ho: fs, dns.lookup, crypto, zlib.
  • Mimo pool (přímo přes OS): síťové/HTTP I/O — sockety.
  • Tvůj JS nikdy neběží v poolu paralelně.

Důsledky (proto to řešíš)

  • Node = skvělý na I/O-heavy služby (API, gateway, real-time). Slabší na number crunching.
  • Těžký CPU výpočet → přesuň mimo hlavní vlákno (Worker Threads, fronta, jiná služba).
  • async/await je všudypřítomné, protože je to způsob, jak „počkat, aniž bys blokoval".
