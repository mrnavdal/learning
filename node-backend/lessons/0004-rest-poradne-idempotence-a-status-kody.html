REST pořádně: idempotence & status kódy

Endpointy už umíš postavit (lekce 03). Teď ať mluví jazykem, který od tebe u pohovoru čekají — správná metoda, správné číslo v odpovědi.

Throughline téhle lekce „REST API" není o tom, že vracíš JSON. Je to dohoda: metoda říká co děláš, status kód říká jak to dopadlo. Když tu dohodu dodržíš, klient (i tvůj budoucí kolega) ví, co čekat, aniž by četl tvůj kód. Když ne, působíš jako junior.

Klíčový pojem: idempotence

Jedno slovo, které u pohovoru rozliší, jestli REST chápeš, nebo jen používáš. Definice (MDN):

Idempotentní metoda Zavoláš ji jednou, nebo pětkrát po sobě se stejnými daty — a stav serveru skončí stejně. Odpověď se lišit může (druhý DELETE vrátí 404), ale stav ne.
Most z Flutteru — proč ti na tom záleží Znáš to z appky: request spadne na timeout, tak ho zkusíš znovu. Otázka zní: je bezpečné retryovat?
  • GET / PUT / DELETE → retry v pohodě. Idempotentní.
  • POST → pozor. Retryem můžeš vytvořit záznam dvakrát (klasika: dvojitá objednávka, dvojitá platba). Není idempotentní.
Idempotence není akademický pojem — je to důvod, proč platební brány řeší Idempotency-Key.

Metody: co dělá, jestli je bezpečná a idempotentní

Safe = jen čte, nemění stav. Idempotent = opakování nevadí. Tohle je celý REST slovník metod v jedné tabulce (MDN Methods):

MetodaDěláSafe?Idemp.?
GETčte✅✅
POSTvytváří nový❌❌
PUTnahradí celý❌✅
PATCHupraví část❌❌*
DELETEsmaže❌✅

* PATCH obecně idempotentní být nemusí (např. „přičti 1" opakováním roste). PUT ano, protože posíláš celý výsledný stav — ať pošleš jednou nebo třikrát, výsledek je tentýž.

Aha — proč PUT je idempotentní a POST ne PUT /users/42 říká „ať uživatel 42 vypadá takhle". Opakuj kolikrát chceš — pořád vypadá takhle. POST /users říká „přidej nového". Opakuj → přidáš víc lidí. Rozdíl je v tom, jestli cílíš na konkrétní URI (PUT), nebo na kolekci (POST).

Status kódy: mapa, kterou musíš mít v hlavě

První číslice je třída. Zbytek jsou detaily. U pohovoru stačí znát tuhle kostru (MDN Status):

TřídaZnamenáKdo to zavinil
2xxPovedlo se—
3xxPřesměrování—
4xxChyba klientavolající (špatný vstup)
5xxChyba serveruty (spadl ti kód)

Pravidlo palce: 4xx = „udělal jsi něco špatně, neposílej to znovu beze změny". 5xx = „moje chyba, klidně to zkus později". Tenhle rozdíl je důvod, proč klienti retryují 5xx, ale ne 4xx.

Ta jedna past, co dělají junioři Vrátit 200 OK na všechno a chybu schovat do těla ({ "error": "..." } se statusem 200). Klient pak nepozná úspěch od selhání bez čtení těla. Status kód je ta informace — používej ho.

Záhadné 409 Conflict — konečně

Slíbil jsem ho už v lekci 03. 409 znamená: „požadavek je v pořádku, ale nemůžu ho splnit, protože by koliduje se současným stavem zdroje" (MDN 409). Dva školní příklady:

// 1) Unikátní e-mail — někdo se registruje na už zabraný e-mail
app.post('/users', (req, res) => {
  if (emailAlreadyTaken(req.body.email)) {
    return res.status(409).json({ error: 'E-mail už je zaregistrovaný' });
  }
  // ...vytvoř a vrať 201
});

Proč ne 400? Vstup není špatný — e-mail je validní. Problém je stav serveru (už tam někdo je). Proto 409, ne 400. Proč ne 500? Nic ti nespadlo, situaci jsi čekal. To je ta jemnost, na kterou se ptají.

// 2) Edit conflict — dva lidi editují ten samý záznam
// Klient poslal PUT s verzí, kterou mezitím někdo přepsal.
app.put('/docs/:id', (req, res) => {
  if (req.body.version !== currentVersion(req.params.id)) {
    return res.status(409).json({ error: 'Dokument někdo mezitím změnil' });
  }
  // ...ulož
});

🖐 Zkus si to — poznej správný kód

Ke každé situaci si nejdřív řekni kód nahlas, pak si ho ověř v kvízu:

POST /users vytvořil uživatele          → ?
DELETE /users/42 proběhl, tělo prázdné   → ?
GET /users/999, ale 999 neexistuje       → ?
POST /users s e-mailem, co už je zabraný  → ?
V handleru padla neošetřená výjimka       → ?

Rychlý kvíz

Zaseklo se něco? Napiš mi. Třeba „kdy 422 místo 400?", „co Location hlavička u 201?", „jak řeším 409 přes DB unique constraint?" nebo „proč PATCH nemusí být idempotentní?". Ptej se průběžně — jsem tvůj učitel.
