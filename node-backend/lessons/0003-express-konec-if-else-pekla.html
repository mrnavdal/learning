Lekce 03 · Express

Express: konec if/else pekla

Stejný server jako minule — ale routing a middleware místo ručního žonglování. Tady tvoje REST API dostane tvar.

Throughline téhle lekce Každá bolest z lekce 02 (ruční routing, parsování /users/42, čtení těla, opakované hlavičky) má v Expressu jednořádkovou náplast. Nic nového koncepčně — jen elegantnější obal nad tím, co už znáš.

Nejdřív: první závislost (a package.json)

Express není vestavěný — musíš ho nainstalovat. Tím se konečně dotkneme package.json, na který ses ptal v úvodním kvízu:

mkdir muj-api && cd muj-api
npm init -y            # vytvoří package.json
npm install express    # stáhne Express + zapíše ho do package.json
Most z Flutteru — package.json vs pubspec.yaml
  • npm install express ≈ flutter pub add. Zapíše závislost do package.json (obdoba pubspec.yaml).
  • node_modules/ ≈ .dart_tool/ / pub cache — stažený kód, necommituje se (patří do .gitignore).
  • Navíc oproti pubspecu: package.json má sekci "scripts" — pojmenované příkazy, které pouštíš přes npm run <název> (třeba npm start, npm test). To pubspec nemá.
  • Verze používají semver (^5.1.0 = „kompatibilní s 5.x"). package-lock.json ≈ pubspec.lock — zamkne přesné verze.

Ten samý server, teď v Expressu

Vzpomeň na if/else peklo z minula. Takhle vypadá v Expressu:

const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('domovská stránka');
});

app.get('/users', (req, res) => {
  res.json([{ id: 1, name: 'Ada' }]);   // res.json = auto stringify + správná hlavička
});

app.listen(3000, () => console.log('http://localhost:3000'));

Všimni si, co zmizelo: žádné if (req.url === …), žádné ruční writeHead, žádné JSON.stringify. (Express docs) Každá cesta je vlastní funkce. A když nic nesedí, Express sám vrátí 404.

Route params — konec ručního parsování

Pamatuješ, jak otravné by bylo z req.url vytáhnout 42 z /users/42? Express to dá zadarmo dvojtečkou:

app.get('/users/:id', (req, res) => {
  res.json({ id: req.params.id });   // GET /users/42 → req.params.id === '42'
});

HTTP metody = CRUD (a oprava z kvízu)

V úvodním kvízu jsi měl POST a PUT lehce prohozené. Narovnejme to — v Expressu je každá metoda vlastní funkce, takže se ti mapování vryje do prstů:

Oprava Řekl jsi „POST zapisuje, PUT vytváří nové entity". Ve skutečnosti je to naopak: POST vytváří, PUT nahrazuje/aktualizuje existující.
MetodaCRUDPříklad
GETRead (čti)app.get('/users/:id')
POSTCreate (vytvoř)app.post('/users')
PUTUpdate (nahraď)app.put('/users/:id')
DELETEDelete (smaž)app.delete('/users/:id')

Paměťová pomůcka: POST jde na kolekci (/users) — „přidej nového". PUT jde na konkrétní kus (/users/42) — „přepiš tenhle". (Proč přesně tak — idempotence a status kód 409 — rozebereme příště v REST lekci.)

Middleware — konečně ta správná definice

V úvodním kvízu jsi middleware popsal dobře („ověří autentizaci/autorizaci"). Teď ti dám přesný tvar. Middleware je funkce se třemi parametry, která sedí mezi příchozím requestem a tvým handlerem:

app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);   // udělej něco (log, auth, …)
  next();   // ⬅ KLÍČOVÉ: pusť request dál na další middleware/handler
});
Ta jedna věc, co lidi pletou next() = „jsem hotový, pošli request dál". Když ho nezavoláš (a ani neodpovíš přes res), request uvázne a klient visí — přesně jako zapomenutý res.end() z minulé lekce. Middleware buď zavolá next(), nebo odpoví (např. res.status(401).json(...) u nepřihlášeného).

A jeden middleware používáš skoro vždy — čtení JSON těla. Vzpomeň, jak jsi v lekci 02 musel tělo POST požadavku číst po kouskách? Tohle je ta náplast:

app.use(express.json());   // naparsuje JSON tělo → naplní req.body

app.post('/users', (req, res) => {
  const newUser = req.body;              // { name: 'Ada' } — už hotové
  res.status(201).json({ id: 1, ...newUser });   // 201 = Created
});

🖐 Zkus si to

# → {"id":1,"name":"Ada"}  se statusem 201
