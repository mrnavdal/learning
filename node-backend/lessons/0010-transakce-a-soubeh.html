Transakce & souběh

Tady se ti slepí dohromady všechno kolem 23505, race conditions a „co když dva requesty naráz". Node zvládá tisíce souběžných requestů (lekce 01) — a všechny sahají do jedné databáze.

Jedna myšlenka, na které stojí celý díl Jeden SQL příkaz je atomický. Dva příkazy za sebou nejsou. Mezi SELECT zůstatku a UPDATE nového zůstatku je škvíra v čase, do které se vejde jiná transakce — a máš lost update. Transakce, izolace a zámky existují proto, aby tu škvíru zavřely.

Transakce a ACID (v kostce)

Transakce = skupina příkazů mezi BEGIN a COMMIT, která platí všechna, nebo žádná. Při ROLLBACK (nebo pádu) se vrátí do stavu před BEGIN. To jsou ta písmena ACID:

  • Atomicity — všechno, nebo nic (viděls u migrací: DDL v transakci se při chybě celé vrátí).
  • Consistency — transakce nechá DB v platném stavu (dodrží constraints z dílu 2).
  • Isolation — souběžné transakce si nelezou do zelí (tenhle díl je hlavně o tomhle).
  • Durability — po COMMIT data přežijou i výpadek proudu.

Problém: lost update

Účet má 100. Dva requesty naráz odečtou po 30 — správně má zůstat 40. Napsáno naivně jako přečti → uprav → zapiš:

const { rows } = await db.query('SELECT balance FROM accounts WHERE id = 1');
const next = rows[0].balance - 30;               // ⬅ tady je ta škvíra
await db.query('UPDATE accounts SET balance = $1 WHERE id = 1', [next]);

Když oba requesty přečtou 100 dřív, než kterýkoli zapíše, oba spočítají 70 a oba zapíšou 70. Jeden odečet zmizel.

Přepínej strategie a sleduj časovou osu (T1 vlevo, T2 vpravo) a DB= zůstatek. U naivní skončíš na 70 (chyba). U atomického UPDATE a zámku na správných 40.

Tři opravy (od nejlepší po nejtěžší)

A) Atomický UPDATE — nech počítat databázi

UPDATE accounts SET balance = balance - 30 WHERE id = 1;

Žádné čtení do aplikace, žádná škvíra — DB provede odečet atomicky v jednom příkazu. Kdykoli jde úprava vyjádřit relativně (balance - 30), tohle je nejlepší volba. Prosté a neprůstřelné.

B) Pesimistický zámek — SELECT … FOR UPDATE

BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- zamkne řádek
-- ...spočítej v aplikaci (T2 zatím čeká)...
UPDATE accounts SET balance = $1 WHERE id = 1;
COMMIT;                                                 -- odemkne

Když musíš mezi čtením a zápisem něco spočítat v kódu, FOR UPDATE řádek zamkne — druhá transakce čeká, dokud první necommitne. „Pesimistický" = předpokládám konflikt a rovnou zamykám.

C) Optimistický zámek — verzovací sloupec

UPDATE accounts SET balance = $1, version = version + 1
WHERE id = 1 AND version = $2;   -- $2 = verze, kterou jsem četl
-- když UPDATE zasáhl 0 řádků → někdo mě předběhl → retry / 409

Pamatuješ version z lekce 04 (a to 409 Conflict)? Tady je jeho mechanika: nezamykám, ale při zápisu ověřím, že se verze nezměnila. Když ano, někdo byl rychlejší → zopakuj, nebo vrať 409. „Optimistický" = počítám, že konflikt je vzácný, a řeším ho, až když nastane. Ideál pro málo kolizní zápisy.

Izolační úrovně (proč to samo neřeší)

Napadlo tě: „nemá to hlídat databáze sama?" Částečně ano — přes izolační úroveň. Postgresí default je READ COMMITTED, který lost update v read-modify-write nezabrání. Nejpřísnější SERIALIZABLE se chová, jako by transakce běžely po sobě — konflikt detekuje a jednu shodí chybou 40001 (serialization_failure), kterou v aplikaci odchytíš a transakci zopakuješ.

Trade-off Vyšší izolace = míň anomálií, ale víc zahozených transakcí a nutnost retry logiky. Nižší = rychlejší, ale hlídáš si souběh sám (atomický UPDATE / zámek). Není „nejlepší" úroveň — je to volba podle toho, co endpoint dělá.
Zpátky k 23505 Ten unique-violation závod z lekce 05 (dva stejné e-maily naráz) je tentýž druh problému: souběh + read-then-write. A řešení bylo stejné jako tady varianta A — nech to na databázi (constraint / atomický příkaz) a odchyť výsledek. Teď vidíš, do jaké rodiny to patří.
