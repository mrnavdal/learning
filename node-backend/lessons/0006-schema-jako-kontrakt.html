Schéma jako kontrakt

V dílu 1 jsi z Node poslal dotaz. Teď o úroveň hlouběji: jak data vypadají a proč dobře navržené schéma udělá půlku tvé validační práce za tebe — a to spolehlivěji, než bys ji kdy udělal v kódu.

Jedna myšlenka, na které stojí celý díl Constraint přesune invariant z naděje do záruky. Aplikační validace je naděje: „doufám, že si každý endpoint vzpomene zkontrolovat, že e-mail je vyplněný a unikátní". DB constraint je záruka: databáze to vynutí u každého zápisu, z jakéhokoli místa, navždy — i z migrace, i z ruky v psql, i z kolegova skriptu. Tohle je ten posun v myšlení, který dělá z „umím SQL" opravdové porozumění.

Typ sloupce je první constraint

Než dojde na pravidla, rozhoduje typ — a volba typu je návrhové rozhodnutí, ne formalita. Pár, které v pohovoru odliší juniora od mid-level:

MístoPoužijProč
penízenumericnikdy float — 0.1+0.2≠0.3, zaokrouhlovací chyby v účtech
častimestamptzukládá v UTC, řeší timezony; timestamp bez tz je zdroj nočních můr
texttextv Postgresu není varchar(n) rychlejší; limit řeš přes CHECK, ne typem
identitabigint/uuidserial (int) přeteče u velkých tabulek; uuid když nechceš hádatelná ID

Špatný typ propašuje bug hluboko do systému. Správný typ ho znemožní — nevalidní hodnota se do sloupce ani nevejde.

Constraints: pět vynucených slibů

CREATE TABLE users (
  id          serial PRIMARY KEY,          -- identita, automaticky unikátní
  email       text   NOT NULL UNIQUE,      -- musí být + žádné dva stejné
  age         int    CHECK (age >= 0),      -- pravidlo na hodnotu
  country_id  int    REFERENCES countries(id)  -- musí ukazovat na reálnou zemi
);
  • NOT NULL — „tohle musí existovat". Bez něj se ti do dat vplíží NULL a každý dotaz na to musí myslet.
  • UNIQUE — „žádné dva řádky nesdílí tuhle hodnotu". Tohle je ten samý constraint, který v dílu 1 vyhodil 23505 → tvoje 409. Teď víš, odkud to číslo je.
  • CHECK — libovolné pravidlo na hodnotu (age >= 0, price > 0). Validace, kterou nejde obejít.
  • PRIMARY KEY — identita řádku. NOT NULL + UNIQUE v jednom, a DB podle něj řádky rychle najde.
  • FOREIGN KEY — referenční integrita. Viz níž, tohle je ten hlubší kus.

Foreign key = tvoje data nemůžou „lhát"

country_id REFERENCES countries(id) znamená: nemůžeš vložit uživatele s country_id = 99, když žádná země 99 neexistuje. DB to odmítne (23503). Bez FK by ti v datech vznikaly „osiřelé" odkazy — řádky ukazující do prázdna.

A druhá strana mince: co když někdo smaže zemi, na kterou ukazují uživatelé? Na to je ON DELETE:

ON DELETE …Co se stane při smazání země
RESTRICTsmazání zakázáno, dokud na ni někdo ukazuje (default, nejbezpečnější)
CASCADEsmažou se i všichni ti uživatelé (pozor — snadno smažeš víc, než chceš)
SET NULLcountry_id se u nich nastaví na NULL
Naděje vs. záruka — proč ne „to ošetřím v kódu" Junior řekne: „validaci e-mailu udělám v Express handleru". Jenže tenhle constraint pak platí jen pro ten jeden endpoint. Přijde druhý endpoint, admin skript, datová migrace — a invariant je pryč. Constraint v DB platí pro všechny zapisovatele bez výjimky. Aplikační validace je navíc (lepší chybová hláška), ne náhrada.

🧪 Sandbox: nech si od DB odmítnout řádek

Tohle je ta samá tabulka users výše. Zkus do ní vkládat řádky — a hlavně je rozbít (tlačítka dole). DB ti řekne, který constraint jsi porušil, jaký má Postgres error kód a jaký HTTP status bys v API vrátil. Tady se ti spojí schéma → error kód → status kód dohromady.

Všimni si toho mostu Každé porušení má svůj Postgres kód (23502, 23505, 23514, 23503) a ty je v API mapuješ na HTTP status (400 / 409 / 422). Odchytáváš je přesně tím try/catch vzorem z dílu 1. Schéma (tenhle díl) + status kódy (lekce 04) + pg (díl 1) jsou teď jeden příběh.

Rychlý kvíz

Cítíš, jak se ti otázky rojí samy? „Jde UNIQUE jen na část řádků (partial index)?" „Co složený primární klíč?" „Kdy se FK kontroluje — hned, nebo až na konci transakce?" To jsou přesně další zastávky v tomhle oblouku, ne slepé uličky. Napiš, kterou chceš dřív.
