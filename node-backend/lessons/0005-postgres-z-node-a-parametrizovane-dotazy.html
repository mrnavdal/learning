Postgres z Node (a proč tě parametry zachrání)

Tvoje API zatím zapomene všechno při restartu. Databáze je paměť. SQL už umíš — tady je server-side delta: jak ho pustit z Node bezpečně.

Throughline téhle lekce Neučím tě SQL — SELECT ... WHERE user_id = 42 jsi v diagnostice napsal správně. Učím tě dvě věci, které SQL-znalý frontendista nemá: jak z Node poslat dotaz do Postgresu a jedno pravidlo, jehož porušení je nejčastější díra v juniorních backendech — SQL injection.

Mluvit z Node s Postgresem: pg

Standardní knihovna je node-postgres (pg). Nainstaluješ ji stejně jako Express:

npm install pg
Most z Flutteru pg je pro databázi to, co pro tebe byl HTTP klient pro API: otevřeš spojení, pošleš dotaz, dostaneš odpověď. A protože je to I/O (jde to po síti do DB), je to async → vrací Promise, dáváš await. Přesně jako Future v Dartu. (Vzpomeň lekci 01: tohle je ta neblokující I/O, kde Node exceluje.) 
const { Pool } = require('pg');
const pool = new Pool();   // čte připojení z env proměnných (PGHOST, PGUSER, …)

app.get('/users/:id', async (req, res) => {
  const result = await pool.query(
    'SELECT * FROM users WHERE id = $1',   // ⬅ $1 je placeholder
    [req.params.id]                        // ⬅ hodnoty zvlášť
  );
  if (result.rows.length === 0) return res.status(404).json({ error: 'Nenalezen' });
  res.json(result.rows[0]);
});

Pool = zásobník znovupoužitelných spojení (otevřít spojení je drahé, tak je držíme). pool.query() vrátí objekt, kde tě zajímá .rows — pole výsledků. Zbytek je SQL, které umíš.

To jediné pravidlo: nikdy nelep vstup do SQL

Všiml sis $1 a hodnot v poli zvlášť? To je parametrizovaný dotaz a je to hráz proti nejklasičtější bezpečnostní díře. Podívej se na ten rozdíl:

// ❌ NIKDY: skládání stringem
const bad = `SELECT * FROM users WHERE username = '${username}'`;

// ✅ VŽDY: parametr — hodnota jde mimo SQL text
const good = await pool.query(
  'SELECT * FROM users WHERE username = $1', [username]
);
Proč to je smrtelné U concat verze zadá útočník do username třeba ' OR '1'='1' --. Tvoje SQL se stane … WHERE username = '' OR '1'='1' --' → podmínka je vždy pravda, -- odkomentuje zbytek. Přihlásí se bez hesla. U parametrizované verze je ten samý text brán jako doslovná hodnota — DB ho nikdy nevykoná jako kód.

🧪 Osahej si to: prolom login (a pak ho oprav)

Vlevo napiš přihlášení. Klikni 💉 Zkus injection a pak přepínej mezi režimy. Ve „String concat" injection projde, v „Parametrizováno" ne — stejný vstup, opačný výsledek. (Uvnitř běží skutečný mini SQL vyhodnocovač, není to podvod.)

Bonus: konečně to slíbené 409 přes DB

Pamatuješ 409 z lekce 04 (zabraný e-mail)? V lekci jsme kontrolovali „ručně". Ale zkontroluj-pak-vlož má race condition — dva souběžné requesty oba projdou kontrolou a oba vloží. Správně: nech unikátnost vynutit databázi a odchyť její chybu (Postgres kód 23505):

try {
  const { rows } = await pool.query(
    'INSERT INTO users(email, name) VALUES($1, $2) RETURNING *',
    [email, name]
  );
  res.status(201).location(`/users/${rows[0].id}`).json(rows[0]);
} catch (e) {
  if (e.code === '23505') {                       // unique_violation
    return res.status(409).json({ error: 'E-mail už existuje' });
  }
  throw e;   // jinou chybu přenech error middlewaru → 500
}

Tohle je učebnicový důvod, proč databáze mají constraints: jsou atomické a jsou zdroj pravdy. Aplikační kontrola je jen zdvořilost navíc.
