Migrace: verzování schématu

Schéma z minula není vytesané do kamene — musí se vyvíjet. Ale máš už data, tým a běžící produkci. Tady je proč se schéma nemění ručně a co dělá migrace na živých datech záludné.

Jedna myšlenka, na které stojí celý díl Migrace je verzování schématu — git pro strukturu databáze. Ne „přihlásím se do psql a dám ALTER TABLE". Migrace je uspořádaná, sledovaná, append-only historie kroků, kterou umí kdokoli deterministicky přehrát z prázdné DB do aktuálního stavu — u sebe na kompu, na CI, na produkci. Stejná struktura všude, žádné „u mě to funguje".

Proč vůbec migrace (a ne ruční ALTER)

Ruční změna schématu má tři fatální díry, které migrace zavírají:

  • Reprodukovatelnost. Kolega, CI i produkce potřebují tu samou strukturu. Migrace = soubory v gitu → každý přehraje stejné kroky.
  • Uspořádání. Migrace jsou očíslované (0001_, 0002_…) a běží v pořadí. DB si v migrační tabulce pamatuje, které už proběhly — takže se každá pustí právě jednou.
  • Historie. Vidíš, jak se schéma vyvíjelo, a k releasu kódu patří odpovídající migrace. Kód a schéma jdou spolu.

Každá migrace má typicky up (aplikuj změnu) a down (vrať ji). down je ale často jen zbožné přání — DROP COLUMN ti data nevrátí. Ber rollback jako záchranu na pár minut po deployi, ne jako spolehlivou cestu zpět.

Ta těžká část: DDL na živých datech

Na prázdné tabulce je migrace triviální. Problém začne, když v tabulce už jsou řádky a běží na ní produkce. Nejklasičtější past:

-- ❌ Naivně: přidat povinný sloupec na plnou tabulku
ALTER TABLE users ADD COLUMN phone text NOT NULL;
-- ERROR: column "phone" contains null values
-- (existující řádky by měly phone = NULL, což NOT NULL zakazuje)

Řešení je rozložit jednu „nemožnou" změnu do tří bezpečných kroků — vzor expand → backfill → contract:

-- ① EXPAND: přidej jako nullable (projde vždy)
ALTER TABLE users ADD COLUMN phone text;

-- ② BACKFILL: doplň hodnotu do existujících řádků
UPDATE users SET phone = '—' WHERE phone IS NULL;

-- ③ CONTRACT: teď už constraint projde
ALTER TABLE users ALTER COLUMN phone SET NOT NULL;
Proč to nejde jedním krokem — hlubší důvod Deploy kódu a migrace nejsou atomické. Chvíli běží stará verze aplikace proti novému schématu (nebo naopak). Kdybys naráz přejmenoval/smazal sloupec, starý kód by spadl. Proto se i rename dělá jako parallel change: přidej nový sloupec → piš do obou → přečti z nového → smaž starý (přes několik deployů). Migrace musí být zpětně kompatibilní s ještě běžící starou verzí.

🧪 Migration runner: rozbij to a pak sprav

Tabulka users má 3 řádky. Klikni 💥 ADD phone NOT NULL (naivně) a uvidíš pád. Pak projdi bezpečnou cestu odshora dolů (nullable → backfill → SET NOT NULL). Sleduj sloupec „Applied" — to je ta verzovaná historie.

Ještě jeden had v trávě: zámky

Některé DDL operace berou zámek, který blokuje zápisy — na velké tabulce i na minuty = výpadek. Nejznámější případ:

-- ❌ zamkne tabulku proti zápisům po dobu stavby indexu
CREATE INDEX idx_users_email ON users(email);

-- ✅ staví index bez zámku zápisů (nesmí být v transakci)
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);

„Migrace proběhla za 3 ms lokálně" ≠ „proběhne bezpečně na produkční tabulce s 50 miliony řádků". (Indexy do hloubky — díl 5.)

Nástroje (koncept > nástroj)

V Node ekosystému node-pg-migrate nebo Knex migrations, s ORM třeba Prisma Migrate. Všechny dělají totéž: číslované soubory, migrační tabulka, up/down, příkaz „domigruj na nejnovější". Až pochopíš proč, nástroj je detail.

Rychlý kvíz

Co ti teď vrtá hlavou? „Jak se řeší migrace, když běží víc instancí appky naráz?" „Co když migrace selže v půlce — je atomická?" „Kdy je bezpečné smazat starý sloupec?" To jsou přesně otázky, co dělají senior backend developera — a další zastávky oblouku.
