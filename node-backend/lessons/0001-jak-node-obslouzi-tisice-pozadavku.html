Lekce 01 · Node runtime

Jak Node obslouží tisíce požadavků naráz

Jedno vlákno, a přesto tisíce klientů. Kouzlo není ve vláknech — je v čekání.

Nejdřív oprava mýtu

Mýtus (tvoje odpověď z kvízu) „Node má víc vláken a používá load balancing — každý požadavek si vezme volné vlákno."

Tohle je model, jak funguje třeba klasický Java/PHP server (vlákno na požadavek). Node to dělá úplně jinak — a právě proto na tuhle otázku u pohovoru padne spousta lidí. Tvůj JavaScript běží v jednom jediném vlákně.

Otázka tedy zní: když je vlákno jen jedno, jak Node zvládne tisíce klientů současně, aniž by se ostatní museli řadit do fronty a čekat?

Klíčová myšlenka Server tráví většinu času čekáním — na databázi, na disk, na síť. Node z toho čekání udělá přednost: místo aby jedno vlákno stálo a zíralo na databázi, pustí ho mezitím obsloužit další požadavky. Vlákno nikdy nezahálí.

Most z Flutteru

Co už znáš z Dartu Dart/Flutter běží na jednom isolate s event loopem. Když napíšeš await http.get(...), UI vlákno nezamrzne — Dart si odskočí dělat jinou práci a k tvému kódu se vrátí, až odpověď dorazí. Node je přesně ten samý princip, jen na serveru místo na UI. Když tohle chápeš u Flutteru, chápeš i Node — jen ten „await" tady čeká na klienty a databáze místo na UI eventy.

Blokující vs. neblokující

Celé to stojí a padá na jednom rozhodnutí: blokující nebo neblokující operace.

// ❌ BLOKUJÍCÍ — vlákno stojí, dokud se soubor nenačte.
//    Po tuhle dobu NEOBSLOUŽÍ nikoho jiného.
const data = fs.readFileSync('big.txt');
console.log('hotovo');

// ✅ NEBLOKUJÍCÍ — Node řekne systému "dej vědět, až to bude",
//    a mezitím obsluhuje další požadavky.
const data = await fs.promises.readFile('big.txt');
console.log('hotovo');

Konvence, kterou si zapamatuj: funkce končící na Sync (např. readFileSync) jsou blokující — na serveru se jim vyhýbáš. (Node docs)

Oficiální dokumentace to ukazuje na číslech: „Každý požadavek trvá 50 ms a z toho je 45 ms databázové I/O, které lze dělat asynchronně. Neblokující operace uvolní těch 45 ms na obsluhu jiných požadavků." Jedno vlákno tak stíhá dělat práci za mnoho.

„Ale nějaká vlákna tam přece jsou…"

Ano — a tady se to jemně komplikuje (a tvoje intuice o vláknech nebyla úplně mimo). Node pod sebou používá knihovnu libuv, která má malý thread pool (ve výchozím stavu 4 vlákna). Ale pozor na dva rozdíly oproti tvé původní představě:

  • Není to vlákno na požadavek. Je to malý sdílený pool — „k workerů, kde k je mnohem menší než počet klientů" (Node docs).
  • Tvůj JS v něm neběží. Pool dělá jen pár konkrétních věcí na pozadí: souborový systém (fs), DNS lookup, crypto, zlib. Síťové I/O (sockety — tedy HTTP požadavky!) jde úplně mimo pool, přímo přes operační systém.

Mentální model, který si odneseš: jedno vlákno na tvůj kód + event loop, který žongluje s čekáním, + malý pomocný pool na pár těžkých operací.

Proč tě to budou ptát u pohovoru Z tohohle modelu plyne všechno ostatní o Node: proč nesmíš na serveru dělat těžké CPU výpočty v hlavním vlákně (zablokuješ event loop a všichni klienti čekají), proč je async/await všudypřítomné a proč Node exceluje na I/O-heavy službách (API, gateway), ale ne na number crunchingu.
