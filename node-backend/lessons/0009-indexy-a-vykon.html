Indexy & výkon

Nejšťavnatější díl oblouku. Jeden CREATE INDEX umí proměnit dvouvteřinový endpoint na dvoumilisekundový — a když pochopíš proč, přestaneš je sázet i tam, kde škodí.

Jedna myšlenka, na které stojí celý díl Bez indexu musí DB pro WHERE email = 'x' přečíst každý řádek a porovnat (sekvenční scan, O(n)). S indexem jde přes seřazený strom a trefí se za ~log n kroků. Pro milion řádků je to rozdíl mezi 1 000 000 a ~20. Index je rejstřík na konci knihy — nečteš celou knihu, skočíš na stránku.

Seq scan vs index scan

Dvě cesty, jak DB najde řádky:

  • Sequential scan — projdi tabulku řádek po řádku, na každém vyhodnoť podmínku. Roste lineárně s počtem řádků. U malé tabulky fajn, u velké katastrofa.
  • Index scan — použij index (seřazenou strukturu) a skoč rovnou na místo. Roste logaritmicky. Dotaz na 10× větší tabulku stojí jen o pár kroků víc.

🧪 Osahej si tu propast

Táhni počet řádků nahoru na milion a sleduj, kolik řádků musí DB „osahat" bez indexu (červená) a s ním (zelená). Pak přepni na LIKE '%x' a uvidíš, kdy index nepomůže vůbec.

B-tree: co umí a co ne

Výchozí index v Postgresu je B-tree — vyvážený seřazený strom. Protože je seřazený, zvládne víc než jen rovnost:

DotazPoužije B-tree?
WHERE email = 'x'✅ rovnost
WHERE age > 30, BETWEEN✅ rozsah (je seřazený)
ORDER BY email✅ už je seřazeno → zadarmo
LIKE 'ada%' (prefix)✅ pevný začátek = rozsah
LIKE '%x.cz' (leading wildcard)❌ nemá kde v stromě začít
WHERE lower(email) = 'x'❌ funkce na sloupci*

* Pokud dotaz volá funkci na sloupci, obyčejný index se nepoužije — potřebuješ functional index CREATE INDEX … ON users(lower(email)). Podobně na fulltext (%x%) se nasazují jiné typy: GIN + trigram.

Jak zjistíš, co DB doopravdy dělá: EXPLAIN

Neháduj — zeptej se. EXPLAIN ukáže plán dotazu, EXPLAIN ANALYZE ho i spustí a změří:

EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'ada@x.cz';

-- Bez indexu:
Seq Scan on users  (cost=0.00..18334 rows=1)  (actual time=210 ms …)
--   ^^^^^^^^ projíždí celou tabulku

-- Po CREATE INDEX ON users(email):
Index Scan using users_email_idx  (cost=0.29..8.31 rows=1)  (actual time=0.05 ms …)
--   ^^^^^^^^^^ skočil rovnou; 210 ms → 0.05 ms

Když v plánu vidíš Seq Scan na velké tabulce u dotazu, který se má trefit přesně — máš kandidáta na index. Tohle je nástroj, kterým se ladí pomalé endpointy v praxi.

Proč tedy neindexovat všechno

Index není zadarmo Každý index je kopie dat navíc, kterou musí DB udržovat při každém zápisu. INSERT/UPDATE/DELETE musí aktualizovat i všechny indexy na tabulce → zápisy jsou pomalejší a index žere místo na disku. Indexuj sloupce, podle kterých filtruješ, joinuješ a řadíš — ne naslepo všechno.

Dva praktické záchytné body, které tě odliší:

  • PRIMARY KEY je automaticky indexovaný, FOREIGN KEY není. Sloupec author_id z dílu 2, po kterém pořád joinuješ, si index musíš přidat sám — jinak je JOIN seq scan. Klasická díra.
  • Na produkci CREATE INDEX CONCURRENTLY (z dílu 3 o migracích) — obyčejný CREATE INDEX zamkne zápisy.
