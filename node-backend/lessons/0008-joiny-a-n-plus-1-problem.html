JOINy & N+1 problém

Foreign key z dílu 2 znamená, že budeš tahat související data. Jak to uděláš rozhoduje, jestli pošleš do DB jeden dotaz, nebo dvě stě. Tady je nejčastější výkonnostní past — a ORM tě do ní navede samo.

Jedna myšlenka, na které stojí celý díl Získat „posty i s jejich autory" jde dvěma cestami se stejným výsledkem, ale propastně jiným počtem dotazů. Naivní cesta škáluje jako 1 + N (roste s daty), správná jako 1 nebo 2 (konstanta). Rozdíl nevidíš na 5 řádcích v devu — uvidíš ho, až produkce načítá 500.

JOIN v jedné větě (SQL už umíš)

JOIN spojí řádky ze dvou tabulek přes klíč. Chceš posty i se jménem autora? Napojíš posts.author_id na authors.id:

SELECT p.title, a.name
FROM posts p
JOIN authors a ON a.id = p.author_id;

Jeden dotaz, hotová data. Drž si tenhle obrázek — je to ta „správná cesta" níž.

N+1: tichý zabiják

Představ si stejný úkol napsaný „objektově", jak tě k tomu svede kód (a hlavně ORM):

const posts = await db.query('SELECT * FROM posts');   // ① jeden dotaz na seznam
for (const post of posts) {
  post.author = await db.query(                        // ② a teď PRO KAŽDÝ post další dotaz
    'SELECT * FROM authors WHERE id = $1', [post.author_id]
  );
}

Máš 1 dotaz na seznam + N dotazů na autory = 1 + N. Pro 5 postů 6 dotazů. Pro 200 postů 201 dotazů — a každý má režii round-tripu na DB. Stránka, co měla trvat 5 ms, najednou trvá vteřinu.

Proč tě do toho ORM navede samo ORM (Prisma, Sequelize, TypeORM) nabízí lazy loading: napíšeš post.author a ono to potají pošle dotaz. Kód vypadá nevinně — jedna tečka — ale v cyklu z toho je N dotazů, které v kódu nevidíš. Proto se N+1 tak snadno propašuje do produkce: nejsou to viditelné await db.query, ale skryté přístupy k property.

Táhni počet postů nahoru a přepínej strategie. U N+1 počítadlo dotazů roste s každým postem; u JOIN zůstane na 1, u Batch na 2 — ať je postů kolik chce. Výsledek (dole) je u všech tří identický.

Dvě opravy — a jejich tradeoff

Jak se N+1 zbavit? Dvě idiomatické cesty, každá s háčkem:

A) JOIN / eager load — 1 dotaz

V ORM se jmenuje include / join / with. Jeden dotaz spojí vše. Háček: u vztahu one-to-many (post má mnoho komentářů) JOIN duplikuje rodičovská data — každý řádek postu se zopakuje pro každý komentář (row explosion). Na to se pak nasazuje GROUP BY / agregace nebo skládání v aplikaci.

B) Batch / IN (…) — 2 dotazy

const posts   = await db.query('SELECT * FROM posts');
const ids     = [...new Set(posts.map(p => p.author_id))];   // unikátní id
const authors = await db.query('SELECT * FROM authors WHERE id = ANY($1)', [ids]);
// spoj v paměti: post.author = authors.find(a => a.id === post.author_id)

Dva dotazy, ať je postů kolik chce, a žádná duplikace. Tohle je přesně princip, který zabalí knihovna DataLoader (batching + cache v rámci requestu) — proto ji potkáš hlavně u GraphQL, kde N+1 hrozí na každém kroku.

Kdy je N+1 vlastně v pohodě Ne vždy je nepřítel. Když je N maličké a fixní (načítáš detail jednoho postu + jeho autora = 2 dotazy) nebo když jsou autoři v cache, N+1 nikoho nebolí. Zabiják je N+1 uvnitř cyklu nad rostoucím seznamem. Nejdřív měř (query log, APM jako datadog), pak optimalizuj — předčasná optimalizace JOINem umí kód zbytečně zamotat.

Jak N+1 vůbec poznáš

  • Query log. Zapni logování SQL (v pg nebo ORM) a koukni, kolik dotazů padne na jeden endpoint. 50 skoro stejných SELECT … WHERE id = ? = N+1.
  • APM. Nástroje jako Datadog/New Relic ukážou „N podobných dotazů" jako typický N+1 signál.
  • Latence roste s velikostí seznamu. Endpoint je svižný na 5 položkách, mrtvý na 500 — klasický otisk.

Rychlý kvíz

Co ti teď naskakuje? „Co když je vztah many-to-many (post má tagy, tag má posty)?" „Kdy přesně JOIN duplikuje řádky a jak to poskládám zpět přes GROUP BY?" „Jak DataLoader ví, které dotazy sloučit?" To jsou další zastávky — napiš, co chceš rozbalit.
